# Vetted Widget Catalog

Every third-party widget on a profile page is a bet on someone else's uptime.
This catalog is the house position: **green-light** widgets are static or
zero-setup and have held up; **red-light** widgets are hotlinked
public instances that rate-limit and go down тАФ never embed them. Everything
dynamic that matters (repo cards, releases, activity, languages) is rendered
by the committed updater from data the user controls, so the page needs few
external widgets.

The red-light rule is absolute for the public instances: a broken-image icon
in the hero position is worse than no widget. If the user insists on one of
those cards, the acceptable form is **self-hosting** тАФ deploy their own
instance (each project's README documents a one-click Vercel deploy with the
user's own PAT) and embed that URL. Otherwise point at the built-in
`featured` managed section, which covers the stats-card use case from data
that can't rate-limit.

## shields.io static badges

```
https://img.shields.io/badge/<LABEL>-<MESSAGE>-<COLOR>?style=<STYLE>
```

- URL-escape inside label/message: `-` тЖТ `--`, `_` тЖТ `__`, space тЖТ `%20`,
  `@` тЖТ `%40`. Color is a hex code without `#` or a named color.
- **Socials (blueprint ┬з2):** `style=for-the-badge`, wrapped in a link:

  ```html
  <a href="https://octocat.dev"><img src="https://img.shields.io/badge/Blog-octocat.dev-1f6feb?style=for-the-badge" alt="Blog"></a>
  ```

- **Star counts on featured cards** are plain text (`тнР 2,048`), not badge
  images тАФ a row of colored chips fights the page. The updater emits them
  from the dossier, only past the `min_stars_badge` gate.

- Logo'd variants exist (`style=flat-square&logo=github`) when a glyph helps;
  don't stack logo + emoji + text in one badge.

## skillicons.dev

```
https://skillicons.dev/icons?i=ts,python,rust,go,docker,kubernetes
```

- Comma-separated icon ids, **cap 16** (taste budget). Optional
  `&theme=dark|light` and `&perline=N`.
- Icon ids are short slugs (`ts`, `js`, `python`, `rust`, `go`, `cpp`,
  `docker`, `kubernetes`, `aws`, `gcp`, `postgres`, `redis`, `react`,
  `svelte`, ...). The full list lives in the skillicons.dev README. A typo'd
  id breaks the **whole grid image** тАФ which is exactly why Phase 5 curls
  every image URL for a 200.

## readme-typing-svg

```
https://readme-typing-svg.herokuapp.com?font=Fira+Code&pause=1000&lines=Line+one;Line+two
```

- `lines` is `;`-separated; spaces become `+`; URL-encode everything else
  (`%2C` comma, `%27` apostrophe).
- Useful knobs: `&color=58a6ff` (hex, no `#`), `&center=true&vCenter=true`,
  `&width=600`, `&repeat=true`.
- This is the hero escape hatch when there's no banner тАФ and it counts as the
  one animated element above the fold, so banner + typing-SVG together is
  always wrong.

## Dark/light image pairs

GitHub respects `prefers-color-scheme` inside `<picture>`. House pattern тАФ
dark in `<source>`, light as the `<img>` fallback, alt text always on the
`<img>`:

```html
<picture>
  <source media="(prefers-color-scheme: dark)" srcset="assets/banner-dark.webp">
  <img src="assets/banner-light.webp" alt="octocat тАФ friendly tentacled tools for developers" width="100%">
</picture>
```

The same pattern serves the snake, whose SVGs live on the profile repo's own
`output` branch (generated by `profile-snake.yml` тАФ `actions.md`):

```
https://raw.githubusercontent.com/octocat/octocat/output/github-snake.svg
https://raw.githubusercontent.com/octocat/octocat/output/github-snake-dark.svg
```

Both 404 until the first workflow run тАФ seed it (Phase 5) before checking
image URLs.

## Alt text

Every `<img>` gets meaningful alt text тАФ the tagline for the banner, the
icon names for the skillicons grid, a one-line description for the snake.
Screen readers read the profile too, and alt text is what renders while a
red-light widget you refused to embed would have been a broken icon.
